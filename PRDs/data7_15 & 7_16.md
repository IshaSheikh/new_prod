## M7_15 Library

The Library module should be modeled around **catalog record -> physical copy -> issue transaction -> return event -> fine workflow**, because the PRD distinguishes book metadata from copy-level circulation and requires immutable issue/return history plus optional, policy-driven fines.[^1]
It also must integrate with platform student identities and parent-child visibility rules, without introducing a separate identity model outside the SIS and RBAC foundation.[^1]

### 1. Entity list

- `library_books` — catalog-level book metadata such as title, author, ISBN, publisher, and category.[^1]
- `library_book_copies` — physical lendable copies tracked individually, because circulation happens at copy level.[^1]
- `library_issues` — active and historical issue transactions tied to a student borrower.[^1]
- `library_returns` — return events tied to a prior active issue.[^1]
- `library_fines` — overdue, damage, or loss fines linked to an issue and optionally to finance.[^1]
- `library_policy_settings` — tenant or school policy for due days, overdue fine rules, and parent visibility.[^1]
- `library_copy_status_history` — immutable history of copy state transitions such as available, issued, overdue, lost, and damaged.[^1]
- `library_issue_events` — immutable workflow history for issue, overdue, return, lost, and closure actions.[^1]
- `library_audit_logs` — auditable create, issue, return, fine, waive, and status-change actions.[^1]


### 2. Table definitions

I am keeping `returns` separate from `issues` because the PRD explicitly models returns as a distinct entity and requires historical issue/return records to remain preserved rather than overwritten.[^1]
I am also using copy-level status and issue-level status separately, because a copy can be unavailable for reasons such as damage or review even after an issue closes.[^1]

### 3. Constraints

The key rules are copy-level uniqueness, one active issue per copy, active-student-only borrowing by default, preserved history, and fine changes that never rewrite issue or return facts.[^1]

- Every table is tenant-scoped, because the PRD requires every book, copy, issue, return, and fine record to belong to exactly one tenant.[^1]
- `library_book_copies.copy_code` must be unique per tenant, because duplicate copy codes are explicitly blocked.[^1]
- A partial unique index should enforce one active issue per book copy, because the same copy cannot be issued twice concurrently.[^1]
- A trigger should block issuing inactive, lost, damaged, under-review, or archived copies, because inactive or lost copies cannot be issued.[^1]
- A trigger should block new issue creation for inactive students unless policy later expands to staff borrowing, because student-focused MVP allows only active students to receive new issues.[^1]
- A trigger should auto-update overdue issue status using `current_date > due_date` or a scheduled job, because overdue reporting must use current date against active issue due date.[^1]
- A trigger should block return creation unless the referenced issue is currently active or overdue, because returns must reference an existing active issue.[^1]
- A trigger should set the copy back to `available` on good return, but to `damaged`, `lost`, or `under_review` when applicable, because return condition changes post-return availability.[^1]
- Fine waiver must require authorized staff and preserve original issue and return dates, because fine settlement cannot retroactively change circulation facts.[^1]

### 6. Query patterns

1. Active and overdue issues for a student, because students may view only their own issue history and due states.[^1]
```sql
SELECT
  i.id,
  b.title,
  c.copy_code,
  i.issued_at,
  i.due_date,
  i.status
FROM library_issues i
JOIN library_book_copies c ON c.id = i.book_copy_id
JOIN library_books b ON b.id = c.book_id
WHERE i.tenant_id = $1
  AND i.student_id = $2
ORDER BY i.issued_at DESC;
```

2. Overdue circulation report, because overdue reporting is a required operational and reporting workflow.[^1]
```sql
SELECT
  i.id,
  s.id AS student_id,
  b.title,
  c.copy_code,
  i.due_date,
  (current_date - i.due_date) AS overdue_days
FROM library_issues i
JOIN library_book_copies c ON c.id = i.book_copy_id
JOIN library_books b ON b.id = c.book_id
JOIN students s ON s.id = i.student_id
WHERE i.tenant_id = $1
  AND i.status IN ('active', 'overdue')
  AND i.due_date < current_date
ORDER BY i.due_date ASC;
```

3. Available copies for catalog search.[^1]
```sql
SELECT
  b.id AS book_id,
  b.title,
  count(*) FILTER (WHERE c.status = 'available') AS available_copies,
  count(*) AS total_copies
FROM library_books b
JOIN library_book_copies c
  ON c.book_id = b.id AND c.tenant_id = b.tenant_id
WHERE b.tenant_id = $1
  AND b.school_id = $2
  AND b.status = 'active'
GROUP BY b.id, b.title
ORDER BY b.title;
```

4. Fine summary by student for parent-linked visibility where enabled.[^1]
```sql
SELECT
  f.student_id,
  count(*) FILTER (WHERE f.status IN ('pending', 'due')) AS open_fine_count,
  sum(f.amount) FILTER (WHERE f.status IN ('pending', 'due')) AS open_fine_amount
FROM library_fines f
WHERE f.tenant_id = $1
GROUP BY f.student_id;
```


### 7. Migration sequence

1. Create enums for book, copy, issue, return, and fine lifecycles.[^1]
2. Create `library_books` and `library_book_copies` first, because copies depend on catalog records.[^1]
3. Create `library_policy_settings` next, because issue and fine validation depend on policy.[^1]
4. Create `library_issues`, then `library_returns`, then `library_fines`.[^1]
5. Create supporting history tables `library_copy_status_history`, `library_issue_events`, and `library_audit_logs`.[^1]
6. Add partial unique indexes, overdue jobs, return-status triggers, and RLS policies on every tenant-owned table.[^1]

### 8. Security-sensitive columns

- `library_issues.internal_note` — internal-only operational notes; must not be exposed to parent or student views.[^1]
- `library_fines.amount`, `library_fines.status`, `finance_invoice_id`, `finance_invoice_line_item_id` — finance-sensitive data; expose only under finance or approved parent visibility policy.[^1]
- `library_audit_logs.old_value`, `library_audit_logs.new_value`, `ip_address`, `user_agent` — auditable security data with restricted access.[^1]
- Student-linked circulation records in `library_issues`, `library_returns`, and `library_fines` are educational records and should be RLS-protected to tenant plus relationship scope.[^1]


## M7_16 HR \& Payroll

For M7_16, the right shape is **staff profile -> attendance and leave -> salary structure snapshot -> payroll run -> payroll item -> payslip -> statutory vault**, because the PRD explicitly separates attendance, leave, payroll, and statutory details while requiring immutable finalized payroll periods and auditable salary changes.[^2]
This module must remain separate from student-facing workflows, use platform user identities where applicable, and protect payroll and statutory fields with stricter controls than general ERP records.[^2]

### 1. Entity list

- `staff_profiles` — tenant-scoped employment records linked to platform users where applicable.[^2]
- `staff_profile_status_history` — effective-dated lifecycle history for draft, active, suspended, reactivated, terminated, and archived states.[^2]
- `staff_attendance` — daily staff attendance, distinct from student attendance.[^2]
- `staff_attendance_corrections` — auditable correction workflow, because attendance corrections must be auditable.[^2]
- `leave_requests` — request and approval workflow for staff leave.[^2]
- `leave_balances` — period-based leave entitlement, usage, and remaining balance.[^2]
- `salary_structures` — effective-dated salary templates or employee-specific salary configuration.[^2]
- `salary_components` — earnings or deduction components with controlled types.[^2]
- `salary_structure_components` — assigned component values within a salary structure.[^2]
- `payroll_runs` — payroll by defined period with lifecycle from draft to finalized.[^2]
- `payroll_items` — one staff member’s payroll result within a payroll run, preserving processed snapshot values.[^2]
- `payroll_item_components` — detailed earnings and deductions snapshot per payroll item.[^2]
- `payroll_adjustments` — explicit post-period corrections instead of silent mutation of finalized payroll.[^2]
- `payslips` — generated and published staff-visible payroll document metadata.[^2]
- `statutory_details` — secured statutory and compliance records per staff profile.[^2]
- `staff_hr_audit_logs` — immutable audit trail across HR and payroll actions.[^2]


### 2. Table definitions

I am separating salary configuration from payroll output because the PRD says payroll records must preserve a salary snapshot at processing time and closed payroll periods must not be recalculated silently.[^2]
I am also isolating statutory records into their own secured table because statutory details are highly sensitive and must be role-protected and audited.[^2]

### 3. Constraints

The hard requirements are active staff lifecycle control, attendance separation from student attendance, overlap-free approved leave, snapshot-based payroll, and secure statutory handling.[^2]

- Every HR and payroll table has `tenant_id`, because the PRD requires every staff profile, leave record, attendance record, payroll record, and statutory detail to belong to exactly one tenant.[^2]
- `staff_profiles.user_id` is optional but unique per tenant when present, because staff must reference platform user identities where applicable, but some staff may operate without a linked login.[^2]
- Staff attendance must be unique per staff and date, because attendance is a daily record and corrections should be auditable rather than duplicate rows.[^2]
- A trigger should block attendance creation for non-active staff, because only active staff may receive active attendance sessions.[^2]
- A trigger should block overlapping approved leave requests for the same staff member, because overlapping approved leave is explicitly disallowed.[^2]
- A trigger should update `leave_balances` only when leave becomes approved, because balances must use effective-dated approved leave rules.[^2]
- Payroll runs should be unique per tenant, school, and period, because payroll must be processed by payroll period.[^2]
- A trigger should block recalculation or silent edits to `payroll_items` and `payroll_item_components` once the payroll run is finalized, because closed payroll periods must not be recalculated silently.[^2]
- Salary structure changes should create new effective-dated rows instead of rewriting prior structures, because salary component changes must be auditable and historical payroll must remain unchanged.[^2]
- Statutory detail changes must be audited and field-level access controlled, because statutory data is explicitly security-sensitive and restricted.[^2]

### 6. Query patterns

1. Staff attendance summary by day and status, because school leaders need attendance summaries and trend visibility.[^2]
```sql
SELECT
  attendance_date,
  status,
  count(*) AS staff_count
FROM staff_attendance
WHERE tenant_id = $1
  AND attendance_date BETWEEN $2 AND $3
GROUP BY attendance_date, status
ORDER BY attendance_date, status;
```

2. Leave approval queue.[^2]
```sql
SELECT
  lr.id,
  sp.staff_code,
  sp.first_name,
  sp.last_name,
  lr.leave_type,
  lr.start_date,
  lr.end_date,
  lr.status
FROM leave_requests lr
JOIN staff_profiles sp ON sp.id = lr.staff_profile_id
WHERE lr.tenant_id = $1
  AND lr.status = 'submitted'
ORDER BY lr.created_at ASC;
```

3. Payroll run review with totals by period.[^2]
```sql
SELECT
  pr.id,
  pr.payroll_period,
  pr.status,
  count(pi.id) AS staff_count,
  sum(pi.gross_amount) AS gross_total,
  sum(pi.deduction_amount) AS deduction_total,
  sum(pi.net_amount) AS net_total
FROM payroll_runs pr
JOIN payroll_items pi
  ON pi.payroll_run_id = pr.id AND pi.tenant_id = pr.tenant_id
WHERE pr.tenant_id = $1
  AND pr.school_id = $2
  AND pr.payroll_period = $3
GROUP BY pr.id, pr.payroll_period, pr.status;
```

4. Self-service payslip retrieval for a teacher’s own linked staff profile.[^2]
```sql
SELECT
  p.id,
  pr.payroll_period,
  pi.gross_amount,
  pi.deduction_amount,
  pi.net_amount,
  p.payslip_status,
  p.published_at
FROM payslips p
JOIN payroll_items pi ON pi.id = p.payroll_item_id
JOIN payroll_runs pr ON pr.id = pi.payroll_run_id
JOIN staff_profiles sp ON sp.id = p.staff_profile_id
WHERE p.tenant_id = $1
  AND sp.user_id = $2
  AND p.payslip_status IN ('published', 'revised')
ORDER BY pr.period_start DESC;
```


### 7. Migration sequence

1. Create all enums for employment, attendance, leave, payroll, salary component, payslip, and statutory states.[^2]
2. Create `staff_profiles` and `staff_profile_status_history` first, because all HR workflows depend on staff identity.[^2]
3. Create operational HR tables `staff_attendance`, `staff_attendance_corrections`, `leave_requests`, and `leave_balances`.[^2]
4. Create payroll configuration tables `salary_components`, `salary_structures`, and `salary_structure_components`.[^2]
5. Create processing tables `payroll_runs`, `payroll_items`, `payroll_item_components`, `payroll_adjustments`, and `payslips`.[^2]
6. Create `statutory_details` and `staff_hr_audit_logs`, then add triggers for overlap checks, finalized-run immutability, and field-level security controls.[^2]

### 8. Security-sensitive columns

- `staff_profiles.date_of_birth`, `official_email`, `mobile_e164` — PII that should be tightly scoped.[^2]
- `salary_structure_components.value_amount`, `payroll_items.gross_amount`, `deduction_amount`, `net_amount`, and `salary_snapshot` — compensation-sensitive payroll data requiring restricted role access.[^2]
- `payslips.file_path` — sensitive staff compensation document location that should never be publicly exposed.[^2]
- `statutory_details.detail_value` — highly sensitive compliance and identity data that should use encryption at rest or application-layer envelope encryption plus strict audit access.[^2]
- `staff_hr_audit_logs.old_value`, `new_value`, `ip_address`, and `user_agent` — security and forensic data with restricted support-only visibility.[^2]
