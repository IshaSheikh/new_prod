# ADR-001: Multi-Tenancy Model — Shared Database with PostgreSQL Row-Level Security

**Status:** Accepted  
**Date:** 2026-01-15  
**Deciders:** Platform Architecture Team

---

## Context

We need to serve multiple K-12 school tenants from a single SaaS deployment. The key requirements are:

1. Complete data isolation between tenants — no tenant can see another's data
2. Cost efficiency at scale — the model must work for 10 schools and 1,000 schools without re-architecture
3. Operational simplicity — one set of migrations, one backup strategy, one monitoring setup
4. Compliance readiness — tenant data isolation must be provably enforced, not just assumed

The three standard approaches to multi-tenancy are:

| Approach | Description | Pros | Cons |
|----------|-------------|------|------|
| **Database per tenant** | Separate PostgreSQL database for each school | Strongest isolation, easy tenant deletion | Very expensive, complex migration management, connection limit issues at scale |
| **Schema per tenant** | Separate PostgreSQL schema for each school | Good isolation | Still expensive, complex migrations, connection sharing issues |
| **Shared schema with RLS** | Single database, single schema, `tenant_id` column on all tables, PostgreSQL RLS policies | Cost-efficient, single migration path, proven at scale | Slightly more complex application logic; misconfiguration could theoretically leak data |

---

## Decision

We will use the **shared database, shared schema with PostgreSQL Row-Level Security (RLS)** approach.

The isolation will be enforced through three defense-in-depth layers:

1. **Domain routing middleware** — every request is bound to a tenant before reaching business logic
2. **JWT claims** — every authenticated token carries `tenant_id` and is validated server-side
3. **PostgreSQL RLS** — every tenant-owned table has a policy that filters records to `current_setting('app.tenant_id')::uuid`

All three layers must pass for any data access to succeed. A failure in the application layer (missing WHERE clause) is caught by RLS at the database layer.

---

## Consequences

### Positive

- **Single migration path:** One `db:migrate` command applies to all tenants simultaneously
- **Cost efficiency:** No N×database overhead; all tenants share connection pool and storage infrastructure
- **Simpler operations:** One PostgreSQL instance (or cluster), one backup strategy, one set of monitoring alerts
- **Performance:** RLS policies are extremely efficient — they compile into the query plan and add negligible overhead
- **Proven pattern:** Used successfully by Supabase, PostgREST, and many production multi-tenant SaaS platforms

### Negative / Risks

- **RLS misconfiguration risk:** If `app.tenant_id` is not set before a query, records from all tenants could be returned (mitigated by the application guard layer and automated cross-tenant tests)
- **Noisy neighbor risk:** A very large tenant with millions of rows could slow queries for smaller tenants (mitigated by partitioning high-volume tables by academic year and using proper indexing)
- **Schema changes affect all tenants:** A migration that adds a column or index runs against the entire table, not just one tenant's data (acceptable trade-off given operational simplicity)

### Mitigations

- Every new table must include `tenant_id NOT NULL` as a required column — enforced in code review
- Automated integration tests include cross-tenant access test cases that must pass before deployment
- High-volume tables (`attendance_records`, `ledger_entries`, `audit_logs`) are partitioned by `academic_year_id` or time period
- Connection pooling uses PgBouncer in **transaction mode** (not session mode) to ensure `SET app.tenant_id` is applied per transaction, not leaked between sessions

---

## Alternatives Considered

**Database per tenant** was rejected because:
- At 1,000 schools, managing 1,000 PostgreSQL databases is operationally impractical for a solo founder or small team
- Connection limits become a severe constraint (PostgreSQL max connections × 1,000 databases)
- Migrations require a fan-out job that is complex to make reliable and consistent

**Schema per tenant** was rejected because:
- PostgreSQL does not natively support cross-schema foreign keys, which would prevent the multi-school user model (users belong to the global schema)
- Still has most of the operational complexity of separate databases without providing full isolation
