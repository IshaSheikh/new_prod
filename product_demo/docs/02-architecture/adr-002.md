# ADR-002: Authentication Strategy — JWT with Refresh Tokens and Tenant-Context Claims

**Status:** Accepted  
**Date:** 2026-01-20  
**Deciders:** Platform Architecture Team

---

## Context

The platform needs an authentication strategy that:

1. Works across web (Next.js), mobile (Flutter), and API consumers
2. Carries tenant context in the token to enable tenant resolution without a database lookup on every request
3. Supports immediate session revocation when a user is deactivated, a tenant is suspended, or a role changes
4. Supports multi-school users (a teacher at two schools must be able to select which school to access)
5. Is stateless where possible to support horizontal scaling

The options considered were:
- **Session-based authentication** (server-side sessions stored in Redis/database)
- **JWT-only stateless tokens** (no server-side storage)
- **JWT + Refresh Token with server-side session record** (hybrid approach)

---

## Decision

We will use **JWT Access Tokens + Refresh Tokens with server-side session records** in the `auth_sessions` table.

### Token Structure

**Access Token (JWT, short-lived: 15 minutes)**

```json
{
  "sub": "user-uuid",
  "tenant_id": "tenant-uuid",
  "membership_id": "membership-uuid",
  "roles": ["teacher"],
  "permissions": ["attendance.mark", "grades.view"],
  "session_id": "session-uuid",
  "iat": 1234567890,
  "exp": 1234567890
}
```

**Refresh Token (opaque, long-lived: 30 days)**
- Stored as a hashed value in `auth_sessions.refresh_token_hash`
- Revocable by updating `auth_sessions.status = 'revoked'`

### Token Refresh Flow

```
Client: POST /auth/refresh (with refresh token)
Server:
  1. Lookup session by refresh token hash
  2. Check session status = 'active' and not expired
  3. Check tenant status (suspended tenants = blocked)
  4. Issue new access token + new refresh token (rotation)
  5. Update auth_sessions with new refresh_token_hash
```

### Multi-School User Flow

When a user belongs to multiple tenants:

```
POST /auth/login
  → Returns: user info + list of tenant memberships (no access token yet)

POST /auth/select-tenant { tenantId }
  → Validates membership
  → Issues access token with tenant_id claim
  → Creates auth_session record
```

This ensures users actively select their working school context before receiving data-bearing tokens.

---

## Consequences

### Positive

- **Immediate revocation:** Access tokens expire in 15 minutes. Refresh tokens are revocable via `auth_sessions.status`. Tenant suspension invalidates all sessions by setting `status = 'revoked'` on all sessions for that `tenant_id`.
- **Horizontal scaling:** Access token validation is stateless (verify JWT signature only). No Redis or session store needed for normal request flow.
- **Tenant context in token:** No extra database lookup per request to resolve tenant context — it's embedded in the token and validated by the tenant guard.
- **Role/permission caching:** Current roles and a subset of permissions are embedded in the access token. Since access tokens are short-lived (15 minutes), role changes are reflected within 15 minutes without active revocation.
- **Session visibility:** Users can see all active sessions (devices) and revoke individual sessions — a requirement for the security and portal modules.

### Negative / Risks

- **Permission staleness:** If a School Admin removes a teacher's role, the teacher's current access token (valid for up to 15 minutes) still carries the old permissions. This is accepted as a known trade-off — for critical permission changes, full session revocation can be triggered.
- **Token size:** Embedding permissions in the access token increases token size. We limit the permissions embedded to the most frequently checked ones; fine-grained permission checks for sensitive operations always go to the database.
- **Refresh token rotation complexity:** Implementing secure refresh token rotation (detect replay attacks) adds implementation complexity but is required for mobile clients.

### Mitigations

- The `auth_sessions` table allows forced revocation of all sessions for a user or tenant when critical changes occur (deactivation, suspension, role removal)
- Access token expiry is set to 15 minutes — a short window that limits exposure from stolen tokens
- Sensitive operations (role changes, subscription cancellation, financial reversals) always check the database for the latest permission state, bypassing the token cache
- Refresh token rotation detects replay attacks: if an old refresh token is presented after rotation, all sessions for that user are revoked

---

## Out of Scope for MVP

The following authentication features are explicitly deferred to Phase 2:
- SSO / SAML / OAuth Enterprise Identity Providers
- Multi-Factor Authentication (MFA)
- Biometric authentication
- Single Sign-On across multiple tenant schools for the same user

These features are noted in the architecture but will not be implemented in V1. The schema supports future MFA through the `tenant_settings.enforce_mfa_for_admins` flag, but the enforcement logic is not built.
